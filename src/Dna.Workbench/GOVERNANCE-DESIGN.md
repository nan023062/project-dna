# Dna.Workbench 治理架构设计 (Governance Design)

> 状态：Active
> 最后更新：2026-04-04
> 适用范围：`src/Dna.Workbench/Governance`

## 1. 核心哲学：对抗 AI 架构熵增

AI Agent 最大的弱点是**“局部极优，全局盲视”**。为了快速完成当前的 `vibe working`（需求开发），AI 倾向于走捷径。随着迭代次数增加，必然会出现：

- **上帝模块**：单一职责破产，所有功能塞进一个模块。
- **知识孤岛**：重复造轮子，缺乏公共下沉。
- **架构漂移**：依赖穿透、循环依赖，破坏 DAG（有向无环图）。
- **契约失效**：代码改了，但 `identity.md` 的接口声明没更新。

因此，**治理（Governance）不是让 AI 随便看看代码，而是一次有范围、有证据、有法则的“架构 GC（垃圾回收）”**。在 Workbench 中，治理本质上是一种“以修复架构为目标”的特殊需求任务。

## 2. 治理的频率与范围分级

绝对不能每次都全局治理，否则会导致 Token 爆炸和 AI 焦点丢失。治理必须基于频率和体量进行分级：

### 2.1 高频治理（如：每日 / 触发式）

- **范围**：**基于记忆的增量扫描**。Workbench 会读取当天的 `Memory` 和 `Session` 记录，**仅提取发生过修改的模块及其直接依赖**作为治理范围。
- **目标**：
  - 清理当天的 Session 垃圾，把有价值的临时发现升级为长期 Memory。
  - 检查刚刚修改的代码是否超出了该模块的 `dna.summary`（职责说明）。
  - 检查新增的 public 方法是否同步到了 `Contract` 中。
- **触发时机**：每天定时，或发生“跨模块修改超过 2 个责任域”时立即触发。

### 2.2 低频治理（如：每周 / 节点式）

- **范围**：**全局拓扑骨架**。不看具体代码实现，只看整个项目的模块节点、依赖连线和契约声明。
- **目标**：
  - 检查全局依赖 DAG 是否健康，是否存在 A -> B -> A 的循环依赖。
  - 检查**层级倒挂**（低 `layerScore` 模块反向依赖高 `layerScore` 模块）。
  - 跨模块去重（DRY），发现平级业务模块的相似逻辑，建议下沉到基础库。
- **触发时机**：每周的“周治理”，或发生重大架构重构前。

### 2.3 未来演进：基于体量的分级治理

随着项目模块树的体量增加，全局治理也会变得过于庞大。未来将支持**按子树（Sub-tree）治理**：

- 比如单独对 `Program/Business` 部门进行治理。
- 比如单独对 `Program/Framework` 技术底座进行治理。

## 3. 治理上下文：Workbench 的“体检报告”

当 Agent 调用 `ResolveGovernance(ScopeId)` 时，Workbench 的角色是 **“架构 Linter”**。它不改代码，而是返回一份 `GovernanceContext`（体检报告），包含：

1. **拓扑快照 (Topology Snapshot)**：目标范围内的模块树与合法的依赖连线。
2. **违规与坏味道 (Violations & Smells)**：
  - **依赖越权**：实际代码的引用超出了 `dependencies.json` 的声明。
  - **循环依赖**：检测到的闭环。
  - **层级倒挂**：违反 `layerScore` 引力法则的依赖。
3. **记忆与契约漂移 (Memory & Contract Drift)**：
  - 尚未合并到官方文档的 `#decision`, `#lesson`。
  - 契约与实现不符的警告。

## 4. 治理法则：Agent 的“宪法”

Agent 拿到体检报告后，必须遵循以下硬性法则来生成修复任务（这些规则将通过 Prompt 强注入给 Agent）：

- **法则 1：DAG 铁律**。依赖必须是有向无环图。遇循环依赖，必须通过提取接口（依赖倒置）或下沉公共逻辑解开。
- **法则 2：引力法则**。依赖只能从高 `layerScore` 指向低层。绝对禁止底层感知上层，必须用事件总线或泛型解耦。
- **法则 3：单一职责**。模块行为超出 `summary` 描述时，必须拆分。
- **法则 4：契约优先 (Contract First)**。外部调用的方法必须在 `identity.md` 声明，否则视为非法暴露，应改为 internal/private。
- **法则 5：DRY 下沉**。发现跨模块重复逻辑，必须剥离下沉到 `Technical` 或 `BaseLib`。

## 5. 治理的执行闭环

1. **收口**：Agent 调用 `ResolveGovernance` 获取体检报告。
2. **拆解**：Agent 根据报告和“宪法”，叹了口气，把问题拆成一个个具体的重构 Task（如“拆分 Payment”、“提取 Util”）。
3. **执行**：Agent 老老实实调用 `StartTask` 拿锁，在受限上下文中改代码，最后 `EndTask` 释放锁并沉淀知识。
