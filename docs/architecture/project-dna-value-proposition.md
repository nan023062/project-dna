# 工程 DNA 价值定位与竞品分析

> **状态**：v1.0 — 2026-03-28
> **作者**：李南

---

## 一、市场全景

2026 年的 AI Agent 知识/记忆相关产品可以分为四个赛道：

```
                    通用 ←────────────────→ 项目专用
                      │                        │
         ┌────────────┼────────────────────────┼───────────┐
  会话/  │  Mem0       Zep                     │           │
  个人   │  Memstate   Prism                   │ Pensieve  │
  记忆   │                                     │ CLAUDE.md │
         ├─────────────────────────────────────┤           │
         │                                     │           │
  项目/  │           （空白地带）                │ GitNexus  │
  团队   │                                     │ Codegraph │
  知识   │       ★ Project DNA ★               │ lxDIG     │
         │                                     │           │
         └─────────────────────────────────────┴───────────┘
                  知识+治理                     纯代码结构
```

**Project DNA 占据的是「项目级 + 知识治理」这个独特位置。**

---

## 二、四条赛道的竞品详析

### 赛道 A：通用 Agent 记忆层

这些产品解决「AI 失忆」问题——让 Agent 跨会话记住信息。

| 产品 | 核心能力 | 存储 | 定位 |
|------|----------|------|------|
| **Mem0** | 分层记忆（会话/用户/组织）、记忆压缩、Graph Memory | 云端 SaaS | 通用记忆层，对接各种 LLM 框架 |
| **Zep** | 时序知识图谱（Graphiti）、跨会话综合、<200ms 检索 | 云端/自托管 | 企业级 Agent 记忆 |
| **Memstate MCP** | 版本化结构记忆、冲突检测、O(1) token 增长 | SaaS | MCP 兼容的结构化记忆 |
| **Prism MCP** | Mind Palace、时间旅行、多 Agent 同步 | 本地 SQLite | 本地优先的 Agent 记忆 |

**它们的共同特点**：
- 以**对话/会话**为中心——记住用户说了什么、偏好什么
- **通用目的**——不关心你在做什么项目、项目结构是什么
- **平铺记忆**——没有模块、依赖、层级等结构概念
- 不理解**项目文件系统**——不知道哪个文件属于哪个模块

**与 DNA 的本质区别**：它们记住的是「人说了什么」，DNA 记住的是「项目是什么」。

### 赛道 B：代码结构分析

这些产品解决「AI 不懂代码结构」问题——让 Agent 理解跨文件依赖。

| 产品 | 核心能力 | 分析粒度 | 定位 |
|------|----------|----------|------|
| **GitNexus** | 六阶段代码索引、知识图谱、跨文件依赖解析 | 函数/类 级别 | 代码知识图谱 MCP |
| **Codegraph** | 函数级依赖图、增量重建、影响分析 | 函数级别 | 代码依赖分析 + CI 质量门 |
| **lxDIG MCP** | 动态智能图、混合检索（Graph+Vector+BM25） | 符号级别 | 代码智能 + Agent 记忆 |
| **codebase-memory-mcp** | 极致性能（Linux 内核 3 分钟索引）、66 语言 | 文件/符号 | 高性能代码索引 |

**它们的共同特点**：
- 以**代码语法**为中心——通过 Tree-sitter AST 解析理解代码
- 粒度是**函数/类/符号**——关心 `class Foo` 调用了 `class Bar`
- **纯代码**——不理解美术资源、策划配表、设计文档
- **无知识沉淀**——知道依赖关系但不记录约束、教训、规范
- **无组织概念**——不知道哪些代码属于哪个团队

**与 DNA 的本质区别**：它们分析的是「代码怎么调用」，DNA 建模的是「项目怎么组织」。

### 赛道 C：个人开发者知识

这些产品解决「个人经验复用」问题。

| 产品 | 核心能力 | 形态 | 定位 |
|------|----------|------|------|
| **CLAUDE.md** | 项目约定文件、自动记忆 | 文本文件 | Claude Code 内建 |
| **Cursor Rules** | .mdc 规则文件、glob 触发 | 配置文件 | Cursor 内建 |
| **Pensieve** | Skill 文件、四层模型（MUST/WANT/HOW/IS）、自动 self-improve | Markdown + 脚本 | 个人知识笔记本 |

**它们的共同特点**：
- **单人使用**——不支持团队共享
- **纯文本**——无语义检索、无图结构
- **平铺知识**——无模块、依赖、层级
- **绑定特定客户端**——CLAUDE.md 只能用于 Claude Code

**与 DNA 的本质区别**：它们是「个人笔记」，DNA 是「团队知识中台」。

## 三、Project DNA 的独特价值

### 3.1 没有任何竞品同时具备以下五个能力

| 能力 | Mem0 | Zep | GitNexus | Codegraph | Pensieve | CLAUDE.md | **DNA** |
|------|------|-----|----------|-----------|----------|-----------|---------|
| 项目结构建模 | ✗ | ✗ | △ 代码级 | △ 代码级 | ✗ | ✗ | **✓ 组织级** |
| 多工种覆盖 | ✗ | ✗ | ✗ | ✗ | ✗ | ✗ | **✓ 程序+美术+策划** |
| 知识沉淀与检索 | ✓ | ✓ | ✗ | ✗ | △ 基础 | △ 基础 | **✓ 四通道** |
| 架构治理 | ✗ | ✗ | ✗ | △ CI 门 | ✗ | ✗ | **✓ 治理引擎** |
| 团队协作 | ✓ 组织记忆 | ✓ | ✗ | ✗ | ✗ | ✗ | **✓ 多人共享** |
| 跨模块协作建模 | ✗ | ✗ | ✗ | ✗ | ✗ | ✗ | **✓ CrossWork** |
| 多角色视角 | ✗ | ✗ | ✗ | ✗ | ✗ | ✗ | **✓ Interpreter** |

**DNA 的核心差异化：它是唯一一个把「组织结构」「知识记忆」「架构治理」统一在同一个图里的系统。**

### 3.2 价值公式

```
Project DNA 的价值 =
    代码结构分析（GitNexus 能做的）
  + 知识记忆层（Mem0 能做的）
  + 项目规范管理（CLAUDE.md 能做的）
  + 架构治理（SonarQube 能做的）
  + 多工种覆盖（没人做的）
  + 跨职能协作建模（没人做的）
  ────────────────────────────────
  统一在一个图模型里，通过 MCP 对外服务
```

### 3.3 一句话定位

> **Mem0 让 Agent 记住用户，GitNexus 让 Agent 看懂代码，Project DNA 让 Agent 理解整个项目。**

## 四、核心价值场景

### 场景 1：Agent 修改代码前知道「什么不能碰」

| 没有 DNA | 有 DNA |
|----------|--------|
| Agent 不知道模块边界，随意跨模块修改 | `begin_task` 返回约束和依赖，Agent 知道改 A 不能影响 B |
| Agent 不知道团队规范，写出不合规代码 | 节点 Knowledge 自带约束规范 |
| Agent 改完不知道影响了谁 | 依赖图告诉 Agent 哪些下游模块可能受影响 |

**竞品对比**：GitNexus 能告诉 Agent「函数 A 调用了函数 B」，但不能告诉它「改函数 A 时不能违反 GC 规范、不能破坏和 VFX 模块的契约」。

### 场景 2：多工种团队的知识共享

| 没有 DNA | 有 DNA |
|----------|--------|
| 策划改了配表，程序不知道 | CrossWork 知识记录了配表和代码的契约 |
| 美术出的资源不合规格 | 美术模块节点自带规格约束 |
| 新人不知道项目规范 | `dna init` + 模板即可获得全部上下文 |

**竞品对比**：所有竞品都只关心程序员。DNA 是唯一覆盖程序+美术+策划+QA 的知识系统。

### 场景 3：架构演化可追溯

| 没有 DNA | 有 DNA |
|----------|--------|
| 架构决策散落在各处（文档、聊天记录、脑子里） | 架构决策记录在图节点的 Knowledge 中 |
| 不知道为什么当初这样设计 | 每个节点有 `Lessons`：为什么这样做、踩过什么坑 |
| 架构腐化无人察觉 | 治理引擎自动检测：环路→建议合并、关键节点→提示稳定性 |

**竞品对比**：没有任何竞品提供「架构治理 + 知识沉淀」的组合。

## 五、潜在风险与应对

### 风险 1：「太重了，不如用 CLAUDE.md」

**现实**：个人开发者觉得跑服务太麻烦，一个 Markdown 文件就够了。

**应对**：
- 提供 `GitNativeBackend`——纯文件模式，不需要跑服务，类似 Pensieve
- `dna init` 5 分钟上手，门槛要和 `create-react-app` 一样低
- Scanner 自动生成 DNA，用户不需要手动填写

### 风险 2：「代码分析不如 GitNexus/Codegraph 深」

**现实**：DNA 的粒度是模块/文件夹级别，不做函数级 AST 分析。

**应对**：
- 这不是同一个赛道——DNA 关注的是「项目认知」而非「代码语法」
- 可以和 GitNexus/Codegraph **互补**：DNA 提供宏观结构和知识，它们提供微观代码分析
- Scanner 插件机制允许集成第三方代码分析结果

### 风险 3：「通用记忆层（Mem0/Zep）已经很成熟」

**现实**：Mem0 融资过亿，Zep 有企业客户，它们的记忆能力确实强。

**应对**：
- DNA 不和它们竞争「记忆」——DNA 的记忆引擎是服务于图结构的，不是通用聊天记忆
- 长期可以**集成**：DNA 管项目知识，Mem0/Zep 管对话记忆，互不冲突
- DNA 的差异化在「图」不在「记忆」

### 风险 4：「只适合游戏团队，通用性不够」

**现实**：当前代码里只有 `GameProjectAdapter`，例子都是游戏场景。

**应对**：
- 这正是 Template + Scanner 生态要解决的问题
- 核心模型（统一节点 + DAG + CrossWork）是完全通用的
- 第一批模板覆盖 2-3 种主流技术栈（Unity、React、Spring Boot）即可证明通用性

## 七、总结

### Project DNA 解决的核心问题

> **AI Agent 修改项目文件时缺乏全局认知。**
>
> 现有工具让 Agent 记住了对话（Mem0）、看懂了代码语法（GitNexus）、
> 学到了个人习惯（CLAUDE.md）——但没有人告诉 Agent：
> 这个项目的组织结构是什么？哪些模块之间有依赖？改这个文件需要遵守什么约束？
> 这次修改会影响哪个团队的哪些模块？程序和策划、美术之间有什么协作契约？
>
> **Project DNA 就是这个「全局认知」。**

### 差异化定位

```
Mem0 = Agent 的海马体（记住对话）
Zep = Agent 的长期记忆（记住用户）
GitNexus = Agent 的视觉皮层（看懂代码）
CLAUDE.md = Agent 的便签纸（个人备忘）

Project DNA = Agent 的前额叶皮层（理解项目全局、规划行动、做出判断）
```
